# Module Promotion Readiness

This checklist shows the evidence expected before a candidate can become an official reusable Velvet module.

It is guidance only. Final promotion decisions remain with maintainers inside the private Module Lab.

## Evidence checklist

- [ ] Purpose is singular and reusable.
- [ ] Inputs and outputs are documented.
- [ ] Authority and forbidden actions are explicit.
- [ ] Hardware access is isolated behind an adapter.
- [ ] Configuration is validated.
- [ ] Failure and degraded behavior are tested.
- [ ] Fixture data is deterministic.
- [ ] Supported Python and runtime versions are named.
- [ ] Receipt behavior is documented where required.
- [ ] Compatible Velvet surfaces are listed.
- [ ] Body-specific assumptions are absent or isolated.
- [ ] Source provenance and authorship remain visible.
- [ ] Known limitations are recorded.

## Test evidence

Promotion normally requires unit tests, contract tests, failure-path tests, compatibility evidence, and hardware-in-loop evidence when hardware claims are made.

## Review outcomes

A candidate may be promoted, held for more work, retained as reference material, or declined with the reason recorded.

A public request or a merged lab candidate is not official by itself. Official status begins only after maintainer review and recorded promotion.
